# Story 生成可靠性与故事锚点重构

- Req ID: REQ-20260409-generation-reliability-refactor
- Status: Planned
- Created At: 2026-04-09 Asia/Shanghai
- Updated At: 2026-04-09 Asia/Shanghai

## 背景

基于 `docs/reports/REPORT-20260409-old-throne-live-review.md` 对线上项目 `旧日王座` 的实际排查，当前系统已经暴露出一类比“字段是否补齐”更上游的问题：

- 总导在真实 Provider 上 100% fallback
- 故事核心模块虽已开始结构化，但线上主读路径尚未切到新结构
- 章节生成缺少稳定锚点，导致角色身份、POV、剧情推进容易漂移
- 录入链路、读取链路、生成链路没有形成强闭环

当前更大的系统瓶颈，不是“模块数量不够”，而是“有效约束不足且不可观测”。

## 目标

- 把章节生成链路从“多个松散模块的可选拼接”，收敛为“强制锚点包 + 可验证生成”。
- 让总导在真实 Provider 下的成功 / fallback 模式可区分、可追踪、可解释。
- 让章节在生成前具备最小可用的故事锚点：章纲、POV、必出人物、剧情推进点。
- 把当前系统从“功能平台”推进到“稳定生产线”。

## 范围内

- AI 总导真实兼容性治理与 fallback 语义显式化。
- 章节生成前的 `readiness` 校验和锚点包组装。
- 章节级锚点的统一读写入口。
- 写作链路的可观测性增强。
- 章节页 / 写作页对生成就绪度和锚点的展示。
- 与现有核心模块重构和总导层的衔接，不另起第二套故事模型。

## 范围外

- 新增向量检索、Embedding RAG、长期记忆系统。
- 再扩一批新业务模块，如势力树、任务树、情感树等。
- 重做聊天式写作面板交互。
- Big Bang 式推翻现有 Outline / Plot / Character / Chapter 模块。

## 约束与默认假设

- 优先收敛复杂度，不继续堆更多概念层和业务层。
- 优先复用当前核心模块重构结果，不引入与其平行的新故事主链路。
- 生成链路缺少关键锚点时，系统应明确告警，必要时阻止生成，而不是静默降级。
- 如果真实 Provider 工具调用仍不稳定，必须显式暴露当前是 `fallback`，不能在 UI 上制造“总导已成功工作”的错觉。
- 第一阶段更关注“生成一致性”和“可观测性”，不是先追求最复杂的 Agent 自主性。

## 验收标准

- 在真实项目样本中，连续多章生成不再出现主角姓名 / 身份明显漂移。
- 章节生成前，系统能明确给出 `readiness` 结果，至少覆盖：
  - 章纲
  - POV
  - 必出人物
  - 剧情节点
- 写作页和章节页都能看出当前一次生成是：
  - 总导成功决策
  - 还是启发式 fallback
- 生成记录可回溯当时使用的锚点包和模块选择结果。
- 章纲到章节的绑定关系清晰，不再让卷纲直接充当章级写作约束。

## 待确认问题

- `chapter brief` 第一阶段是作为派生对象即时组装，还是需要持久化。
- 生成就绪度第一阶段是否先以“强提示 + 可继续”为主，还是直接启用硬阻断。
- opening 类章节是否允许 plot / causality 作为可选锚点，而不是必填锚点。
- 当前 `knowledge_document` 是否要在第一阶段就拆成 canon / cache 两类。
