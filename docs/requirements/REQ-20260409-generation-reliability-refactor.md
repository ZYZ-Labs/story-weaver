# Story 生成可靠性、减法交互与结构化沉淀重构

- Req ID: REQ-20260409-generation-reliability-refactor
- Status: In Progress
- Created At: 2026-04-09 Asia/Shanghai
- Updated At: 2026-04-10 Asia/Shanghai

## 背景

基于 `docs/reports/REPORT-20260409-old-throne-live-review.md` 对线上项目 `旧日王座` 的真实排查，以及 2026-04-10 的进一步讨论，当前系统的核心问题已经进一步收敛：

- 总导在真实 Provider 上 100% fallback
- 新结构虽然部分落地，但线上主读路径并未真正切到新结构
- 章节生成缺少稳定锚点，导致角色身份、POV、剧情推进容易漂移
- 前台暴露给普通作家的字段过多，其中大量字段意义不明、触发机制不透明
- 录入链路、读取链路、生成链路没有形成“对话输入 -> 摘要确认 -> 结构化保存 -> 稳定生成”的闭环
- 从代码实现看，当前 `/api/ai-writing/*` 主链虽然带有 `plan / write / check / revise` 工作流日志，但本质上仍是多阶段、无状态的 `generate` 流水线；现有聊天能力只是背景参与上下文来源，不是正文生成时的持续会话。

当前更大的系统瓶颈，不是“模块数量不够”，而是：

- 有效约束不足
- 作者认知负担过高
- 结构化数据没有以作者自然表达为入口产生

同时需要明确一个判断：

- 当前系统在有大纲、剧情、章节、因果这些上游约束的前提下，并不属于完全开放式创作，更接近“半开放式创作”。
- 真正的问题不是“完全没约束”，而是这些约束没有稳定转化成当前创建对象和生成动作的即时判断依据。

## 目标

- 把章节生成链路从“多个松散模块的可选拼接”，收敛为“最小必要锚点包 + 可验证生成”。
- 把作者前台交互从“直接填写大量结构字段”，收敛为“对话 / 引导式输入 + 摘要确认”。
- 让系统把自然语言输入整理成可复用的结构化数据，而不是要求作者一开始就理解内部字段设计。
- 让总导在真实 Provider 下的成功 / fallback 模式可区分、可追踪、可解释。
- 把当前系统从“功能平台”推进到“稳定生产线”，同时降低普通作家的使用门槛。
- 为开发者 / 专家用户增加“进度预测 / 节奏建议”能力，在创建大纲、剧情和章节时，根据已有数据推断当前更适合的推进阶段与对象类型，但该能力必须是按钮触发、建议性质，而不是自动执行。
- 明确 `chat` 与 `generate` 的职责边界：前者负责采集、澄清、补问和整理摘要，后者负责在确认后的锚点与约束下产出最终章节文本。

## 范围内

- AI 总导真实兼容性治理与 fallback 语义显式化。
- 章节生成前的 `readiness` 校验和锚点包组装。
- 作者对话式 / 引导式采集流程，以及其后的摘要确认与结构化写回。
- 项目级、人物级、章节级摘要沉淀策略。
- 聊天会话、摘要建议和最终生成之间的交接边界治理。
- 章节页 / 写作页对生成就绪度、锚点和当前决策模式的展示。
- 现有复杂字段的减法与分层：普通模式只暴露必要字段，专家模式保留深层结构。
- 大纲、剧情、章节创建时的“进度预测 / 节奏建议”能力，默认只在专家模式或开发者模式中暴露。
- 与现有核心模块重构和总导层的衔接，不另起第二套故事模型。

## 范围外

- 新增向量检索、Embedding RAG、长期记忆系统。
- 再扩一批新业务模块，如势力树、任务树、情感树等。
- Big Bang 式重做整个前端站点。
- 让 AI 在无确认前提下自动改写全部项目结构。

## 约束与默认假设

- 结构化数据应更多作为系统内部稳定层和专家层，而不是普通作者的前台主交互。
- 普通作者默认只需要回答“这是谁、这一章要发生什么、不能违背什么、这章写到哪”这类问题。
- 专家模式默认不打扰普通作者，但任何作者都可以主动开启；开启后只增加可见字段和手工控制能力，不改变默认生产线。
- 对话式输入后的结构化写回必须经过摘要确认，不能直接静默覆盖实体数据。
- 生成链路缺少关键锚点时，系统应明确告警，必要时阻止生成，而不是静默降级。
- 如果真实 Provider 工具调用仍不稳定，必须显式暴露当前是 `fallback`，不能在 UI 上制造“总导已成功工作”的错觉。
- 第一阶段优先解决“生成一致性 + 降低作者负担 + 可观测性”，不是先追求最复杂的 Agent 自主性。
- 正文生成链路应保持结果导向的 `generate` 执行；聊天链路主要承担采集、澄清和摘要整理，不应把“是否存在聊天会话”伪装成“正文生成一定具备自动补救能力”。

## 验收标准

- 普通作者在不理解大量底层字段的前提下，也能完成项目基础设定和章节准备。
- 至少通过项目级 / 人物级 / 章节级摘要确认流程，把对话或自然语言输入稳定写回结构化数据。
- 在已有大纲、剧情、章节和因果数据的项目中，系统可以对“当前即将创建的大纲 / 剧情 / 章节更像主线推进、支线展开、转折、收束、伏笔还是兑现”给出可解释的预测建议。
- 章节生成前，系统能明确给出 `readiness` 结果，至少覆盖：
  - 章纲或章节 brief
  - POV
  - 必出人物
  - 剧情推进点
- 写作页和章节页都能看出当前一次生成是：
  - 总导成功决策
  - 还是启发式 fallback
- 在真实项目样本中，连续多章生成不再出现主角姓名 / 身份明显漂移。
- 作者默认界面中需要手工填写的字段数量明显下降，并且每个保留字段都能解释其实际生效方式。
- 写作页和日志能明确说明：当前是“基于确认后摘要和锚点的一次生成流水线”，而不是“正文正在持续多轮聊天会话中”。
- 系统能区分并展示三类产物：
  - 聊天过程内容
  - 待确认摘要包
  - 最终生成文本

## 待确认问题

- 对话式采集第一阶段是复用现有聊天能力，还是单独做轻量“整理助手”入口。
- 结构化摘要写回时，采用“逐字段确认”还是“整包确认”。
- 项目级 / 人物级 / 章节级摘要优先落到现有实体字段，还是部分写入 `knowledge_document` 的 canon 类来源。
- `readiness` 第一阶段是否先以“强提示 + 可继续”为主，还是直接启用硬阻断。
