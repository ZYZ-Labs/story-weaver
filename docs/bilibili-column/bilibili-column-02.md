# **为什么我决定不再继续修 Prompt，而是把小说写作系统重做成一个平台**

> 这是上一篇之后的第二篇。  
> 如果说第一篇更多是在讲“这个项目是怎么长出来的”，那这一篇想讲的，是它为什么必须被重做。  
> 不是因为它不能用，而是因为我越来越清楚地意识到：  
> **继续在原来的路上修，只会越修越复杂。**

------

## 它的问题，已经不是“再调一调 Prompt”能解决的了

前面一段时间，我其实一直在做一件事：

> 修生成链路。

比如：

- 补章节锚点
- 加总导
- 加检查
- 加修订
- 加摘要确认
- 加读者揭晓边界

这些事情没有错。  
甚至可以说，它们都是必要的。

但做到后面，我越来越明显地感觉到一件事：

> **我们碰到的不是单点问题，而是架构问题。**

总导不稳定，不能简单理解成“Prompt 写得不够好”。  
写手容易漂，也不能简单理解成“模型太笨”。  
字段太多，也不只是“前端表单丑”。

真正的问题是：

- 上下文主要还是靠应用层临时拼接
- 状态没有被稳定外置
- 用户界面暴露的是结构字段，而不是创作摘要
- 一个模型经常同时承担“决定写什么”和“把它写出来”

换句话说，这个系统已经不再像一个创作平台。

它更像一个：

> **勉强能工作的 Prompt 编排器**

这就是为什么，后面再继续一味地修 Prompt，我反而越来越没信心。

------

## 我后来意识到，结构化不能消失，但不能继续直接暴露给人

这个认知对我影响很大。

以前我会本能地觉得：

> 要稳定，就得更结构化。

这句话只说对了一半。

因为结构化确实重要。  
没有结构化，AI 系统最后一定会漂。

但问题是：

> **结构化不等于让作者去填一堆字段。**

人物、世界观、物品、技能、因果、剧情，这些东西如果全都做成字段墙，最后会同时压垮三种人：

- 作者
- AI
- 专家用户

作者脑子里本来就是模糊形象、感觉、关系和片段。  
AI 更擅长从自然语言里做归纳，不擅长先天按字段思考。  
而专家虽然理解结构，也不代表愿意一直当数据录入员。

所以后来我和 Codex 反复讨论之后，最终把这件事定成了一个硬原则：

> **用户只看摘要，结构下沉到系统内部。**

也就是：

- 前台默认展示摘要
- 结构化字段给系统、MCP、状态机和编排层用
- 用户只确认“摘要变化”，不确认原始 JSON

这个转变看起来像是交互优化，但其实背后是整个系统形态的变化。

------

## 为什么我要把它改成一个“状态驱动平台”

后来我越来越确定，这个系统后面不会只处理：

- 世界观
- 人物
- 剧情
- 章节

它迟早还要处理更多运行时东西：

- 背包
- 技能
- 关系变化
- 态度变化
- 好感度
- 世界观影响
- 已揭晓与未揭晓信息
- 镜头之间的交接状态

这些东西如果没有一个统一的承载方式，后面一定会变成“各写各的”。

但我又不想搞一个万能大基类，把所有业务都塞进去。  
那样只会变成另一个灾难。

最后我们讨论出来的方案，是一个比较克制的中间路线：

> **用一个统一协议壳来承载多个信息切面。**

这个协议壳后来被我们定成了：

> `StoryUnit`

它不是业务万能对象，也不是超大总表。  
它更像是给 MCP/LSP 和多 session 工作流用的统一读写单元。

在这个单元下面，再分成不同的 facet：

- Summary
- Canon
- State
- Relation
- Reveal
- Execution

这一步一旦成立，后面很多事情就都顺了。

------

## 我们后面不再打算让一个模型同时负责“决定”和“表演”

这也是一次非常大的思路转向。

以前总觉得，一个足够强的模型，理论上应该能：

- 理解全局
- 决定节奏
- 再把正文直接写出来

这在理想状态下没错。  
一个成熟作家在脑子里，确实可能同时完成这几件事。

但放到工程里，我越来越觉得：

> **你不能指望一个模型在一个 session 里一直同时把这几件事都做好。**

所以后面系统准备改成四个 session：

- 总导
- 选择器
- 写手
- 审校器

更进一步，章节本身也不再是一把写完。

而是：

- 章节总导先拆出骨架
- 骨架再拆成若干镜头
- 每个镜头单独决定、单独写、单独交接

这个变化很关键。

因为一旦镜头状态可以被外置、被快照、被承接，系统就不再只能靠 Prompt 自己“记住前文”。

它开始真的像一个有状态的故事引擎。

------

## MCP 和 LSP 的引入，不是锦上添花，而是迟早要来的基础设施

我后来越来越确信，MCP 和本地状态协议服务不是“高级玩法”，而是必需品。

没有它们的时候，很多上下文只能这样处理：

- 应用层拼出来
- 直接塞 Prompt
- 模型用完就丢

这会直接导致两个问题：

1. Prompt 越来越大
2. 状态越来越不可追踪

所以后面我准备把这一层正式拆出来：

- `MCP Server`
  - 负责稳定工具读取和受控写回
- `Local Story Protocol Server`
  - 负责本地状态、快照、patch、event 和运行时读模型

到那时候，总导不需要再背那么多包袱。

写手也不需要靠运气去猜：

- 当前人物身上有什么
- 技能能不能用
- 读者已经知道什么
- 上一个镜头到底停在哪

这些东西都应该变成可读取、可校验、可回放的结构化状态。

------

## 这已经不是一个 MVP 了

这一点我现在看得很清楚。

如果是半年前，我可能还会本能地想：

> 要不先做个 MVP，跑起来再说。

但现在已经不是这个阶段了。

一方面，系统已经不再只是“试试看 AI 能不能写小说”。  
另一方面，我们也已经知道它真正的难点在哪里了。

所以后面这件事不能再按“补几个功能”推进。

它必须被当成一个长期平台工程来做：

- 要有主需求
- 要有主计划
- 要有架构文档
- 要有实施计划
- 要有进度文档
- 每次开发前还要先拆本轮详细计划

这个过程会很重。  
也会很慢。  
甚至很可能会长到上千个工程文件，几十份甚至更多文档。

但我现在反而觉得：

> **只有承认它是个大工程，它才有可能真正做成。**

------

## 我为什么还愿意继续做

说到底，是因为我越来越觉得这件事值得做。

不是因为“AI 写小说”这个噱头本身。  
而是因为它背后，其实在逼着我思考一个更大的问题：

> 一个长链路、强上下文、强状态依赖的创作系统，到底应该怎么被搭起来？

这个问题，放在小说里成立。  
放在游戏里成立。  
放在很多复杂 Agent 系统里，其实也同样成立。

所以我现在已经不太把它看成一个单纯的写作项目了。

它更像是一个入口。  
一个让我认真去做“状态驱动创作平台”的入口。

而接下来真正困难的，也不是某一条 Prompt。

而是：

- 架构能不能立住
- 状态能不能收住
- 文档能不能跟上
- 系统能不能在复杂度上升之后，依然保持可维护

如果这些问题能被慢慢解决，那这个项目才算真正开始。

------

## 结尾

上一篇我写的是，它是怎么长出来的。  
这一篇想写的是，它为什么必须被重做。

对我来说，这其实不是推倒重来。

更像是终于承认：

> 它已经长到必须进入下一阶段了。

接下来，我会继续把新的架构文档、计划和实现慢慢补起来。  
这条路不会短，但至少方向比以前清楚很多了。
