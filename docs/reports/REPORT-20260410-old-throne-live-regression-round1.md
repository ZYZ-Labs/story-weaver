# 旧日王座线上回归报告（第一轮）

日期：2026-04-10

目标项目：`旧日王座`（projectId=`28`）

目标地址：`https://home.silvericekey.fun:41202/`

## 本轮结论

本轮线上回归确认：第一章的问题不是“首章完全写崩”，而是“首章已完成钩子后，继续拟生成的续写记录发生了目标漂移和收束失败”。

更准确地说：

- 已接受的第一章草稿 `ai_writing_record.id=27` 质量基本可用，能完成“退役者收到邀请函并决定赴约”的章节目标。
- 后续拟生成记录 `ai_writing_record.id=33` 才是明显的低质量样本。
- 该续写记录不是单纯文风问题，而是生成链路没有约束“本次续写应该补什么、停在哪里、如何收束”。

## 线上证据

### 1. 第一章已接受稿可用，但续写稿失控

章节：

- `chapterId=31`
- 标题：`退役者的邀请函`

写作记录：

- `id=27`
  - `status=accepted`
  - `generated_chars=2453`
  - 主体完成了主角低谷、邀请函触发、回归决定三个关键节点。
- `id=33`
  - `status=rejected`
  - `original_chars=2453`
  - `generated_chars=3493`
  - 文本从“已经完成本章钩子”之后继续外扩，开始大段补旧战史、战术文件、论坛舆论、设备说明等内容。
  - 结尾直接停在“他”字，属于明显未完成收束。

结论：

- 用户感受到的“没头没尾”，对应的是 `id=33` 这条续写记录，不是 `id=27` 的首版章节草稿。

### 2. 续写目标没有被局部化

当前写作链路对 `continue` 的默认任务语义仅为：

- “承接现有正文自然续写，不要重复已经写过的句子”

但没有补上这些必要约束：

- 本次续写只解决一个局部目标
- 本次续写预计新增多少字
- 本次续写停在什么动作点
- 如果上一版已经完成本章钩子，则禁止继续扩写成下一章内容

这会导致模型在“第一章已经完成钩子”之后，仍继续往下补设定、补回忆、补世界观，最后把章节边界写散。

### 3. 总导给的是卷级目标，不是首章续写目标

第一章最新总导决策 `decisionId=9` 的硬约束里，仍带着整卷级别目标，例如：

- 完成世界观落地
- 建立主角回归动机
- 引出旧时代公会影响力
- 完成新手区成长线
- 建立第一轮势力冲突
- 埋下后续公会争霸主线

这些目标对“第一章首次续写”来说过大。

结果就是模型会把本该只做局部补强的一次续写，写成整卷铺垫的继续展开，章节边界自然会发虚。

### 4. 首章上下文本身还存在潜在 canon 冲突

第一章锚点与章节信息中，主角名已经稳定为：

- `林沉舟`

但总导工具链拿到的卷纲上下文里，仍然出现旧主角名：

- `主角马柯作为旧时代老玩家回归`

这说明项目内部 canon 还没有完全清洗干净。

虽然第一章正文最终没有直接写出“马柯”，但这种上游设定残留会持续污染后续决策和生成。

### 5. 项目整体一致性仍处于危险区

线上一致性接口：

- `GET /api/projects/28/story-consistency`

返回核心结果：

- `score=0`
- `status=critical`
- `directorFallbackRate=75.0`
- `traceCoverageRate=25.0`
- `blockedReadinessCount=3`

其中更严重的后续问题集中在：

- `chapterId=33`：缺失 POV 锚点、缺失章纲绑定、总导 fallback
- `chapterId=34`：缺失 POV 锚点、缺失章纲绑定、总导 fallback

并且正文主角名已出现漂移：

- `林沉舟 -> 林风 -> 林渊`

所以第一章让人觉得“不够稳”，并不是孤立现象，而是整条章节链稳定性不足的前兆。

## 根因判断

本轮我给出的根因排序如下：

1. `continue` 写作模式缺少“局部目标 + 收束点 + 新增范围”约束。
2. 总导在首章续写场景下仍注入过大的卷级目标，导致章节任务失焦。
3. 项目上游 canon 未完全清洗，卷纲中仍保留旧主角名 `马柯`。
4. 后续章节锚点缺失，进一步放大了用户对整条故事线“没头没尾”的主观感受。

## 建议动作

### P0

- 给 `continue` 模式增加局部续写协议：
  - 本次只完成一个局部目标
  - 明确新增字数目标
  - 明确收束动作点
  - 若本章主钩子已完成，则优先提示“建议新建下一章或改用 expand”

- 给生成结果增加最小完整性检查：
  - 禁止结尾停在半句
  - 检查末段是否完成动作或情绪收束
  - 若明显截断，则自动判定需要 revision

### P1

- 调整总导决策策略：
  - 当 `writingType=continue` 且当前章节已形成明显结尾钩子时，总导只下发局部续写目标，不再携带整卷目标。

- 清理 `旧日王座` 项目内遗留 canon：
  - 把卷纲、大纲、剧情中残留的 `马柯` 全部排查并统一为当前主角名。

### P2

- 在写作页显式提示：
  - 当前章节是否已经达到“章节可收束”状态
  - 当前操作更适合 `continue`、`expand` 还是“新建下一章”

## 直接判断

如果现在继续拿当前线上版本直接做“第一章继续拟生成”，大概率还会再出一批“内容有料但不成章”的文本。

不是模型不会写，而是系统没有把“续写的边界”讲清楚。
