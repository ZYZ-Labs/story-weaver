- `readiness=blocked` 已真实生效。

## 结果

### A. Blocked readiness 已真实阻止生成

对 `chapter 32 / 33 / 34` 直接调用 `POST /api/ai-writing/generate`，全部返回 `HTTP 400`。

典型返回：

- `《算法少女苏晚》 尚未满足生成前置条件：缺少章纲或章节 brief，无法稳定约束本章生成范围。请先补齐章节摘要、POV 或人物锚点后再生成。`
- `《训练赛首胜》 尚未满足生成前置条件：缺少章纲或章节 brief ...；缺少 POV 人物锚点。；缺少人物锚点，至少需要 POV 或必出人物。请先补齐章节摘要、POV 或人物锚点后再生成。`

判断：

- 这一项 P0 已通过线上验证。
- 之前“readiness 已 blocked 但仍继续生成”的问题已收口。

### B. 第一章从零开始写，开场不再像中段硬切

流式回归中，chapter 31 的真实正文开头为：

`傍晚六点十七分，林沉舟关掉了办公室的灯。`

随后继续交代：

- 写字楼、走廊、电梯、地铁站等现实场景
- 林沉舟退役两年后的身体与生活状态
- 规律克制、近乎仪式化的日常

判断：

- 这次开场已经是标准冷启动写法。
- 读者定向是成立的。
- “像默认读者已经知道上文、从章节中段直接切入”的老问题，这一轮没有复现。

### C. 结尾完整性硬检查已真实生效

chapter 31 的本轮生成最终没有返回成功记录，而是直接 `HTTP 400`。

错误消息：

- `生成结果未通过硬性完整性检查：正文结尾停在未完成句子、缺少自然收束，不能直接作为可提交结果。请重试或调整本章锚点后再生成。`

并且本轮失败后：

- `ai_writing_record` 没有新增新记录
- 项目最近记录顶部仍然停留在旧的 `id = 35`

判断：

- 坏稿已经不会再落库。
- 这一项 P0 也通过了线上验证。

### D. 但第一章空章首稿仍容易在结尾阶段失败

本轮对 chapter 31 至少做了三次真实请求：

- 使用 DTO 默认 `maxTokens = 500`
- 显式指定 `maxTokens = 700`
- 显式指定 `maxTokens = 1200`

这三次都被“结尾完整性硬检查”挡回。

其中流式抓到的 `500` 样本正文末尾停在：

- `说是书房，其实只是卧室隔出来的一个小角落。一张书桌，一把人体`

判断：

- 当前线上版本已经不会保存这种半句坏稿。
- 但第一章从零起稿的成功率仍不够高，尤其是“自然收束”这一项还不稳定。
- 这不再是“中段切入”问题，而是“首稿收尾稳定性”问题。

### E. 总导真实 Provider 仍 fallback

本轮流式日志仍然显示：

- `总导已回退到启发式决策，本轮仍继续走生成流水线`
- `模型在工具调用后仍未返回最终决策结果`

项目级一致性报告仍为：

- `score = 0`
- `status = critical`
- `directorFallbackCount = 4`
- `directorFallbackRate = 100.0`

判断：

- 总导兼容性问题仍未收口。
- 当前线上系统依旧主要依赖 fallback decision pack。

## 额外发现

### 1. `AIWritingRequestDTO` 的默认 `maxTokens` 仍为 `500`

代码位置：

- `backend/src/main/java/com/storyweaver/domain/dto/AIWritingRequestDTO.java`

这会导致“没有显式传 `maxTokens` 的直接 API 调用”天然更容易被截断。

### 2. 前端写作页默认值是 `700`

代码位置：

- `front/src/views/writing/WritingView.vue`

但本轮 `700` 样本依旧未通过结尾完整性检查，说明问题并不只是一处默认值，而是“首稿长度预算 + 收束策略”仍需继续收紧。

### 3. 错误消息格式仍有标点重复

当前线上错误消息里仍能看到：

- `范围。。`
- `锚点。；`

这是消息拼接层的小问题，不影响阻断逻辑，但建议顺手修掉。

## 本轮结论总结

- 好消息：
  - `blocked readiness` 已真实阻止生成
  - 空章起稿的开场方向已明显变对
  - 半句坏稿已不会再落库
- 剩余问题：
  - 第一章从零起稿的首稿收尾稳定性仍不足
  - 总导在真实 Provider 上仍 100% fallback
  - 错误消息格式还有标点瑕疵

## 建议下一步

1. 先处理“首稿收尾稳定性”
   - 提高首章 / 空章起稿的默认 `maxTokens`
   - 在 writer prompt 里显式要求本轮必须写到可自然停住的位置
   - 在 revise prompt 里补一句“如果已接近 token 上限，优先完成当前段收束，不要继续扩展新场景”
2. 再继续打总导真实兼容
   - 重点看真实 DeepSeek 在工具调用后的最终返回格式
3. 顺手修掉错误消息标点拼接
