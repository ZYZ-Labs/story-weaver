- Scope:
  - 面向 UI 手工测试
  - 覆盖人物、世界观、章节三条摘要工作流主链
  - 兼顾普通模式与专家模式

## 怎么使用这份测试数据

这份数据不是拿来一次性导库的。

正确用法是：

- 复制其中一条“想法”
- 粘贴到普通模式对话框
- 观察 AI 追问、摘要草稿、预览和写回结果

## 推荐测试环境

### 环境 A：练新增

推荐新建一个单独的练习项目，例如：

- 项目名：`摘要工作流练习场`

原因：

- 新增测试容易留下对象
- 独立项目最方便反复试错

### 环境 B：练编辑

推荐使用现有项目：

- `旧日王座`
- `projectId = 28`

推荐练这个现成对象：

- 第 31 章
- 当前原摘要：
  - `两年沉寂后，主角收到旧战队邀请，命运再次启动。`

这个对象已经被反复验证过，适合拿来做“编辑现有摘要”的练习。

## 测试路线总览

建议按这个顺序测：

1. 人物新增
2. 世界观新增
3. 章节新增
4. 章节精修
5. 专家模式对比

## 场景 1：人物新增

### 使用入口

- 人物管理
- `说想法新增人物`

### 第一轮输入

```text
我想要一个很油滑的经纪人，表面圆滑，其实特别会算计，和主角以前有合作。
```

### 第二轮可补输入

```text
他现在最想重新绑定主角，因为他觉得主角一旦回归，自己也能重新翻身。
```

### 第三轮可补输入

```text
读者第一次见到他时，应该觉得他很会说话、像朋友，但又有点不可信。
```

### 期望现象

- AI 会继续追问关系、目标或气质
- 右侧会生成人物摘要草稿
- 点击 `看看整理结果` 后会出现：
  - 写回前
  - 写回后
  - 风险提示
- 点击 `确认创建` 后，会新增一个人物对象

### 本轮重点检查

- 摘要是否清楚写出“他是谁”
- 是否写出了“和主角的关系”
- 是否写出了“当前目标”

## 场景 2：世界观新增

### 使用入口

- 世界观管理
- `说想法新增世界观`

### 第一轮输入

```text
这个世界里，职业资格不是考试决定的，而是靠一种叫回声签约的仪式。签约成功后会得到能力，失败后会留下记忆缺口。
```

### 第二轮可补输入

```text
它会直接影响老玩家回归，因为很多旧时代玩家的资格已经失效，想重新进入核心圈子就必须重新签约。
```

### 第三轮可补输入

```text
这个设定最好不要一开始全部讲清，先让读者看到它带来的副作用，再慢慢揭晓规则。
```

### 期望现象

- AI 会追问“影响谁”“为什么重要”
- 摘要草稿会从“设定描述”逐渐变成“设定 + 作用 + 揭晓方式”
- 生成预览后，变化说明会强调这是世界观描述更新
- 确认创建后，会新增一个世界观对象

### 本轮重点检查

- 摘要里是否同时包含“是什么”和“为什么重要”
- 是否出现了读者揭晓顺序

## 场景 3：章节新增

### 使用入口

- 章节管理
- `说想法新增章节`

### 第一轮输入

```text
我想写一章主角和这个经纪人重新见面的戏。先写主角本来不想理他，经纪人又装得很热情，最后停在经纪人抛出一个让主角动摇的条件。
```

### 第二轮可补输入

```text
这章最好让读者先看到主角现在过得并不好，所以他嘴上拒绝，实际上已经开始动摇了。
```

### 第三轮可补输入

```text
这一章不要直接答应合作，停在主角意识到自己其实没有别的路可走。
```

### 期望现象

- AI 会追问“谁出场”“停在哪”
- 摘要草稿会体现：
  - 本章冲突
  - 本章停点
  - 读者第一次知道什么
- 点击 `确认创建` 后，会新增一个章节对象

### 本轮重点检查

- 摘要有没有明确停点
- 有没有把章节写成“正文段落”

如果 AI 直接把摘要写成小说正文，这轮就不合格，应继续改。

## 场景 4：现有章节精修

### 使用入口

- 项目：`旧日王座`
- 章节管理
- 第 31 章
- `摘要优先编辑`

### 当前原摘要

```text
两年沉寂后，主角收到旧战队邀请，命运再次启动。
```

### 普通模式输入

```text
我想让这章摘要更明确一点。先让读者看到林沉舟现在的现实状态，再把邀请函作为触发点抛出来，最后停在他决定赴约。
```

### 期望现象

- AI 返回 `draftSummary`
- 可以继续进入 `看看整理结果`
- 变化预览里，“写回后”比原摘要更具体

### 本轮重点检查

- 是否写出“现实状态 -> 邀请函 -> 决定赴约”的顺序
- 是否没有越界写到下一章

### 注意

如果只是练习，不想污染线上数据：

- 看完 `proposal / preview` 就停
- 不要点 `确认写回`

## 场景 5：专家模式对比

### 用法

在任意一个对象的摘要工作流里：

- 先用普通模式整理到一个可用草稿
- 再点 `切到专家模式`

### 推荐操作

- 选择 `精修摘要`
- 手动把一句关键话改得更明确
- 再生成一次预览

### 重点观察

- 普通模式更适合“从无到有”
- 专家模式更适合“我已经知道哪句话要改”

## API 样例

如果你想直接测接口，可以用下面这些 JSON 结构。

### 1. 普通模式对话采集

接口：

- `POST /api/summary-workflow/chat-turns`

请求体样例：

```json
{
  "targetType": "CHARACTER",
  "projectId": 28,
  "intent": "CREATE",
  "operatorMode": "DEFAULT",
  "messages": [
    {
      "role": "assistant",
      "content": "现在是新增模式。你先说这个人物的大概印象就行。"
    },
    {
      "role": "user",
      "content": "我想要一个很油滑的经纪人，表面圆滑，其实特别会算计，和主角以前有合作。"
    }
  ]
}
```

期望关键返回字段：

- `assistantMessage`
- `draftSummary`
- `pendingQuestions`
- `readyForPreview`

### 2. 生成提案与预览

接口：

- `POST /api/summary-workflow/proposals`

请求体样例：

```json
{
  "targetType": "CHAPTER",
  "targetSourceId": 31,
  "projectId": 28,
  "summaryText": "两年沉寂后，林沉舟仍困在平淡而压抑的现实生活里。一封来自旧战队的邀请函突然打破沉寂，迫使他重新面对自己与旧日赛场的关系，最终停在他决定赴约。",
  "intent": "REFINE",
  "operatorMode": "DEFAULT"
}
```

期望关键返回字段：

- `proposal`
- `preview`
- `preview.beforeSummary`
- `preview.afterSummary`
- `preview.changeSummary`

### 3. 确认写回

接口：

- `POST /api/summary-workflow/apply`

请求体样例：

```json
{
  "proposalId": "替换成上一步返回的 proposalId",
  "confirmed": true
}
```

期望关键返回字段：

- `applied=true`
- `updatedSummary`
- `targetRef`
- `warnings`

## 一轮测试是否算通过

满足下面 5 条，就算这轮通过：

1. 你不用先想字段，也能开始操作
2. AI 至少能追问 1 到 2 个关键问题
3. 能看到摘要草稿
4. 能看到写回前后差异
5. 你知道这次改动到底写回了什么，没有“黑盒写库”的感觉

## 最后提醒

当前版本最适合测试的是：

- 摘要整理能力
- 预览能力
- 写回摘要能力

当前版本不适合拿来要求一步到位的：

- 深层结构自动建模
- 复杂状态流转
- 正文自动修正全部设定

按这个边界测试，结论会更准。
