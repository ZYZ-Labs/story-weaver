- Date: 2026-04-09 Asia/Shanghai
- Scope:
  - 线上站点 `https://home.silvericekey.fun:41202`
  - 项目 `旧日王座`（`project.id = 28`）
  - 对应数据库 `story_weaver`
  - 对应接口：`/api/projects/*`、`/api/world-settings/*`、`/api/ai-director/*`
- Evidence:
  - 线上 API 实际返回
  - MySQL 实际存量数据
  - 当前仓库中的 requirement / plan / progress 文档

## 一句话结论

当前系统的主要问题，不是“总导完全没发挥作用”这么简单，而是：

- 真实 Provider 下总导 100% fallback
- 新结构设计尚未进入线上主读路径
- 章节生成缺少强制锚点
- 录入链路和 AI 读取链路没有真正闭环

结果就是：系统表面上拥有很多模块，但真正进入生成约束的有效结构不够，最终表现为章节一致性漂移、角色锚点丢失、剧情链和因果链更像“事后摘要”而不是“前置骨架”。

### 3. 世界观模块目前是最接近“可被 AI 利用”的模块

## 关键问题

### 1. 总导在线上样本里 100% fallback，真实 tool-calling 没有闭环

观察到的事实：

- `system_config` 中：
  - `ai.director.enabled = true`
  - `director_ai_provider_id = 7`
  - `director_ai_model = deepseek-chat`
- `project_id = 28` 下共有 6 条 `ai_director_decision`
- 6 条记录状态全部为 `fallback`
- 错误一致为：`总导层返回内容不是有效 JSON`

这说明：

- 总导“入口”是开的
- 但真实兼容 Provider 并没有稳定产出当前协议期待的结构化结果
- 线上看到的所谓“总导效果”，本质上仍主要来自启发式兜底，而不是稳定的工具决策

影响：

- 不能把当前章节结果好坏简单归因为“总导已经充分工作”
- 当前真实运行模式，仍然更接近“旧规则聚合 + 总导外壳”

### 2. 新结构开关仍然全部关闭，核心重构尚未成为线上主读路径

观察到的事实：

- `story.refactor.v1.enabled = false`
- `story.refactor.v1.read_new_relations_first = false`
- `story.refactor.v1.enable_outline_tree = false`
- `story.refactor.v1.enable_story_graph = false`

这意味着：

- 当前线上虽然已经有新字段、新关系表、新 VO
- 但业务主路径并没有真正切到“新结构优先”
- 现在更像“结构准备好了，但主流程还在旧路上跑”

影响：

- 设计层面已经变先进，实际运行层面仍然保守
- 用户会感受到系统很复杂，但复杂度并没有有效转化为生成质量

### 3. 章节锚点明显不足，AI 没有被稳定约束住

观察到的事实：

- `chapter_character` 在项目 28 下为 `0`
- `chapter_plot` 在项目 28 下为 `0`
- `chapter 32/33/34` 的 `outline_id` 为空
- `chapter 33/34` 的 `main_pov_character_id` 为空
- 4 个章节的 `requiredCharacterIds` 全为空

这说明：

- 章节并没有形成稳定的“生成前锚点包”
- 即使人物、剧情、大纲存在，也没有强制绑定到具体章节
- AI 在写章时得到的是“可选背景素材”，不是“必须遵守的章节约束”

这是当前线上质量问题的核心根因之一。

### 4. 大纲链路存在错位，章节级骨架没有真正成为章节的直接上游

观察到的事实：

- 大纲共有 3 条：
  - `8` 全书总纲
  - `9` 第一卷：新纪元开服
  - `10` 第一章：新纪元开服
- `chapter 31` 绑定的是 `outline_id = 9`
- 但更接近章节级语义的大纲其实是 `outline 10`
- `outline 10` 反而没有 `generated_chapter_id`，处于“已存在但未真正接入章节”的状态

这说明：

- 现在的大纲树结构存在
- 但“卷纲 -> 章纲 -> 章节”的生成与绑定流程没有完全打通
- 章节 31 实际上吃的是卷级语义，而不是章级语义

影响：

- 第一章能写，但写作约束粒度偏粗
- 后续章节更容易因为上游约束不精确而开始漂移

### 5. 剧情与因果更多是“章节生成后的补录摘要”，不是“章节生成前的约束骨架”

观察到的事实：

- 4 条 plot 全部带 `chapter_id`
- 标题均为“围绕《某章节》的剧情节点”
- `prev_beat_id / next_beat_id` 全为空
- `event_result` 基本为空
- `outline_plot = 0`
- `outline_causality = 0`
- 仅有 1 条因果，且是 `chapter 31 -> story_beat 9`

这说明：

- plot 目前更像是从章节内容回填出来的摘要对象
- causality 也更像点状补充，而不是稳定图谱
- 它们尚未成为“AI 在写章前必须参考的主链路”

影响：

- 名义上的“剧情节点链”“因果链图”在线上样本里还没形成
- 这会直接降低总导和写作层对剧情推进的结构化控制力

### 6. 人物模块有字段，但角色锚定没有真正进入章节写作

观察到的事实：

- 项目里有 3 个主要人物：
  - 林沉舟
  - 苏晚
  - 顾承霄
- `character` 已有 `identity / coreGoal / growthArc / roleType`
- 但：
  - `chapter_character = 0`
  - 章节 `requiredCharacterIds = []`
  - 多章 `mainPovCharacterId = null`

实际效果非常直观：

- 大纲里主角写的是“马柯”
- 第 1、2 章正文主角是“林沉舟”
- 第 3 章正文主角变成“林风”
- 第 4 章正文主角又变成“林渊”

这不是单纯文风波动，而是角色身份锚点失效。

### 7. 知识库当前更像“章节缓存”，容易放大检索噪音

观察到的事实：

- 项目 28 下 4 条 `knowledge_document`
- `source_type` 全是 `chapter`
- 标题分别就是 4 章章节名
- 摘要直接来自章节正文内容

这意味着：

- 当前知识库没有明显区分：
  - 世界规则
  - 稳定人物设定
  - 剧情事实
  - 章节缓存
- 对写作和总导来说，这种知识源很容易把“刚生成出来的漂移文本”再次喂回系统

影响：

- 如果章节已经有轻微漂移，知识检索会继续放大这种漂移
- 知识库没有真正承担“稳定 canon”职责

### 8. 生成追踪链不完整，难以复盘“内容是怎么出来的”

观察到的事实：

- 全库 `ai_writing_record = 32`
- 但 `project_id = 28` 下 `ai_writing_record = 0`
- 同时 `project_id = 28` 下已有章节正文、知识文档和总导决策

这说明至少在当前样本里，以下链路没有完整串起来：

- 生成请求
- 总导决策
- 正文产出
- 章节采纳
- 知识同步

影响：

- 很难精确回答某一章到底是：
  - 通过哪个入口生成的
  - 当时吃了哪些上下文
  - 是否真的经过总导
  - 是否经过采纳链路

这会严重影响线上调试和后续重构验证。

## 根因判断

### 不是素材不够，而是“有效约束对象”不够

系统里确实已经有很多数据，但真正能稳定约束写章的对象还太少。

目前对 AI 最关键的约束，应该至少包括：

- 当前章真正绑定的章纲
- 当前 POV
- 本章必出人物
- 当前章必须推进的剧情节点
- 当前章关联的关键因果
- 允许引用的稳定世界设定

而在线上样本里，这套约束包并没有被强制凑齐。

### 不是单个模块设计错，而是录入链路、读取链路、生成链路没有闭环

当前系统的真实状态是：

- 上层设计想走结构化
- 部分接口也已经返回结构化字段
- 但用户录入动作没有被引导到“必须形成章节锚点包”
- AI 读取时又大量依赖这些可选绑定

于是系统复杂度上来了，但确定性没有同步上来。

### 当前更像“模块平台”，还不是“稳定生产线”

从产品视角看，现在系统已经很像一个功能平台：

- 有世界观
- 有大纲
- 有剧情
- 有因果
- 有人物
- 有章节
- 有总导

但从生产线视角看，还缺 3 个硬环节：

- 生成前校验
- 强制锚点组装
- 生成后可回溯

## 建议的下一轮重构方向

### P0：先修可靠性，不继续扩模块

- 让总导在真实 Provider 上要么成功，要么明确暴露为 `heuristic fallback`
- 给章节生成增加“就绪度检查”
- 未绑定章纲 / POV / 必出人物 / 剧情节点时，不允许默默进入高自由度生成
- 先把“章节生成锚点包”做硬，再谈更多高级功能

### P0：把“章纲 -> 章节 -> AI 生成”收成一条清晰链路

- 章纲必须是章节的直接上游，不再让卷纲直接承担章级约束
- 章节页需要一个非常明确的“本章约束面板”
- 生成前让用户看到本章实际会喂给 AI 的关键锚点，而不是只看到一堆分散模块

### P1：把 plot / causality 从“补充模块”拉回“主线骨架”

- plot 不应该只是“围绕某章生成的摘要”
- 应该是“章节写作前就决定本章要推进什么事件节点”
- causality 不应该只是一条描述性记录
- 应该能表达“这一章推进了哪条因果，兑现了哪条伏笔”

### P1：把知识库拆成稳定 canon 和生成缓存两类

- 世界规则、人物稳定设定、长期事实应该是稳定知识
- 章节生成后的摘要应该是缓存型知识，不应与 canon 同权
- 否则系统会把一次漂移继续放大

### P2：补一致性检查器和线上指标

- 检查人物名漂移
- 检查 POV 漂移
- 检查章节是否跳过预定剧情节点
- 检查总导成功率 / fallback 率
- 检查章节生成是否有完整 anchor snapshot

## 我的判断

我认为这个系统现在确实需要重构，但不应该继续做“大而全”式重构。

下一轮最值得做的，不是再新增一批模块，而是把系统收敛成更少但更硬的几个对象：

- Story Bible
- Volume / Chapter Outline
- Chapter Anchor Pack
- Draft Generation
- Canon / Memory

如果继续沿着“再补一个模块、再补一层关系、再补一个 AI 能力”走，复杂度还会继续上升，但线上质量不一定变好。

相反，如果先把“章节生成锚点”和“真实可观测性”做硬，哪怕总导暂时还是半启发式，系统质量也会明显提升。

## 建议落地顺序

1. 先做生成可靠性与故事锚点重构
2. 再决定是否继续推进当前 Story 核心模块重构剩余步骤
3. 最后再做更深的 AI 总导 / 知识检索增强

原因很简单：

- 现在最影响结果的，不是功能不够多，而是生成约束不够硬
- 先把“写出来不能漂”解决，后面的高级能力才值得继续加
