# AI 写作工作流修订展示缺陷

- Req ID: REQ-20260408-writing-workflow-ui-bug
- Status: Fixed
- Created At: 2026-04-08 Asia/Shanghai
- Updated At: 2026-04-09 Asia/Shanghai

## 背景

当前 AI 写作后端工作流已包含“准备上下文 -> 背景整理 -> 规划 -> 写作 -> 检查 -> 修订”多阶段流程，且在修订成功时会通过流事件把正文替换为修订后的最终版本。

用户反馈前端对“校准/检查后再修订正文”的支持不够清晰，需要先做代码定位并形成 bug 文档。

## 目标

- 定位当前前端在 AI 写作工作流阶段展示上的缺陷。
- 明确是“后端未发事件”还是“前端未正确消费/展示事件”。
- 形成后续修复的最小范围和优先级判断依据。

## 范围内

- `WritingView` 写作中心。
- `ChapterListView` 章节页初稿助手。
- `AIProcessLogPanel` 工作流日志面板。
- AI 写作流式事件的前后端契约。

## 范围外

- 模型本身是否真的完成了高质量修订。
- 聊天式写作面板的交互问题。
- 非流式同步生成接口的表现。

## 约束与默认假设

- 优先复用现有流事件协议，除非现有协议无法表达关键状态。
- 缺陷定位以当前代码为准，不先假设是 Provider 或模型行为不稳定。
- 修复目标应优先解决“用户看不见流程语义”和“用户不知道最终内容已被修订稿替换”。

## 验收标准

- 用户能看出当前工作流是否进入了“准备上下文”“背景整理”“检查”“修订”等阶段。
- 当修订结果通过 `replace` 事件覆盖原草稿时，前端有明确可见反馈，而不是仅静默替换正文。
- 写作中心与章节页两个入口的工作流展示语义保持一致。

## 待确认问题

- 是否只补前端日志/提示文案即可满足预期。
- 是否需要在后端额外增加“修订已应用”这类显式事件，而不仅是 `replace`。
- 是否需要把检查摘要和修订原因展示得更详细，而不是只显示简短日志。
