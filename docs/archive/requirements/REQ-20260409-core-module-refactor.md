# Story 核心模块重构

- Req ID: REQ-20260409-core-module-refactor
- Status: Archived
- Created At: 2026-04-09 Asia/Shanghai
- Updated At: 2026-04-10 Asia/Shanghai

## 背景

`docs/story_weaver_module_refactor_prd_v_1.md` 提出了现有小说创作系统在大纲、剧情、因果、人物、章节上的结构性缺陷。结合当前代码基线核对，问题已经比较明确：

- 大纲仍是“章节附属物”模型，缺少全局/分卷/章节三级层次，也没有真正的树状结构。
- 剧情节点仍主要是自由文本和顺序号，缺少明确的链式连接和事件职责语义。
- 因果关系仍以字符串实体类型和字符串实体 ID 组织，图谱可读但不可稳定级联。
- 人物模块虽然已经支持复用、项目关联和背包，但核心信息仍大量堆在 `attributes` JSON 中，创建成本偏高。
- 章节模块目前仍偏基础 CRUD，无法承担“章节工作台”的职责。

与此同时，AI 写作上下文链路和 AI 总导层已经基本落地，说明上层编排能力已具备雏形，但底层故事数据模型还没有同步升级。这会直接限制后续 AI 上下文命中率、结构化引用能力和 Agent 自动化能力。

## 目标

- 建立稳定的故事数据主链路：世界观 -> 大纲 -> 剧情节点 -> 因果链 -> 章节 -> 正文。
- 让大纲、剧情、因果、人物、章节都具备更清晰的职责边界与结构化字段。
- 把跨模块引用从自由文本 / CSV / 宽松 JSON，逐步升级为稳定的 ID 级引用。
- 降低人物和章节的创建成本，提升长篇小说管理体验。
- 为 AI 总导层、写作上下文聚合和后续 Agent 自动生成提供更稳定的数据基础。

## 范围内

- 大纲、剧情、因果、人物、章节的数据模型调整。
- 对应后端接口、服务、Mapper、SQL 迁移脚本的重构。
- 对应前端管理页与交互流程的重构。
- AI 写作上下文聚合和 AI 总导层对新数据模型的适配。
- 兼容旧数据的迁移、回填、双写/双读与灰度切换方案。

## 范围外

- 彻底重写 AI 总导层或替换现有写作聊天面板。
- 第一阶段引入向量检索、Embedding RAG 或新的大型业务域模块。
- 登录、权限、多租户模型的重做。
- 一次性 Big Bang 式替换所有旧接口和旧字段。

## 约束与默认假设

- 该重构必须分阶段进行，不能打断现有 AI 写作和 AI 总导主链路。
- 旧接口需要保留兼容期，至少在一个完整迁移阶段内可继续工作。
- 优先引入结构化关系表和明确枚举，不继续扩大 CSV / 自由文本字段的使用范围。
- 因果模块涉及多态实体引用，数据库层不强求全量外键，改由应用层做引用校验和删除保护。
- 人物高级资料允许暂时继续使用 JSON 承载，但必须把快速创建和 AI 高频读取字段提升为一等字段。
- 任何迁移不得丢失用户现有内容。

## 验收标准

- 大纲支持 `global / volume / chapter` 三级类型和父子树关系。
- 剧情节点支持明确的事件类型、事件功能和前后链式关系。
- 因果模块支持扩展关系类型、兑现状态、触发模式和上下游链路。
- 人物模块支持“两阶段建角”，并把背包稳定落在角色详情域中。
- 章节模块具备摘要、状态、关联大纲、剧情节点、POV、前后章节等工作台字段。
- AI 上下文聚合与 AI 总导层能读取新结构，并继续兼容旧数据。
- 数据迁移后，旧项目内容可被正确回填和继续使用。

## 待确认问题

- `plot` 是否首阶段继续沿用现有资源名，仅升级字段语义，还是同步改名为 `story_beat`。
- 因果图的可视化视图是否首阶段一起交付，还是先保证数据结构和列表页。
- 人物高级属性最终是继续单 JSON 聚合，还是拆成一张从表。
- 大纲树首阶段是否需要拖拽排序，还是先支持父子挂接和列表/树切换。
