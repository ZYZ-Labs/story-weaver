# Story Weaver 三阶段实施计划

Plan Timestamp: 2026-04-02 21:30 Asia/Shanghai
Status: Completed on 2026-04-03

## 完成情况

- Phase 1 已完成：`front/public/cover.png` 已应用到登录页主视觉区域，保留了原有登录/注册流程与移动端适配。
- Phase 2 已完成：后端已建立 `common / ai / item` 模块化基础结构，并将物品与 AI JSON 生成能力落到新的模块边界，同时保留旧接口路径稳定。
- Phase 3 已完成：项目物品库、角色背包、AI 物品生成、前端背包管理界面与角色背包摘要均已接入。
- 验证已完成：前端 `npm run type-check`、`npm run build` 通过；后端 `mvn test` 全量通过。

## 需求大纲

- 将 `front/public/cover.png` 正式接入前端界面，优先用于登录页主视觉区域。
- 对后端单体项目进行高度模块化重构，保留单体部署方式，但按业务模块和职责层次重新组织代码。
- 新增游戏化物品能力，支持道具、药品、装备等项目级物品定义。
- 新增角色背包管理，支持角色持有、装备、调整数量和移除物品。
- 新增 AI 物品生成能力，可结合项目和角色上下文生成结构化物品数据。

## Phase 1 - Cover 前端应用

### 目标

- 将 `cover.png` 应用于登录页左侧主视觉区。
- 保留当前登录/注册表单行为和路由逻辑不变。

### 实施内容

- 改造 `front/src/views/login/LoginView.vue` 的左侧展示区。
- 引入 `/cover.png` 作为主视觉图层，重组标题、说明和卖点卡片布局。
- 处理桌面端和移动端显示差异，避免图片压迫表单区域。

### 验收标准

- 登录页首屏稳定展示封面图。
- 移动端页面仍可正常登录，布局不溢出。
- 登录、注册、重定向逻辑无回归。

## Phase 2 - 后端结构化处理

### 目标

- 将后端从按技术层平铺的结构，提升为按业务模块组织的高度模块化单体。
- 建立可复用的 AI 能力骨架，为人物属性生成和物品生成共用。

### 实施内容

- 调整 Spring Boot 和 MyBatis 扫描方式，允许模块化分包。
- 以 `common / ai / item` 为优先模块建立 `web / application / domain / infrastructure` 分层，并在现有 `character` 服务中吸收背包摘要与项目内角色聚合逻辑，优先保证接口稳定。
- 将新物品与背包能力直接放入模块化结构，不再继续堆叠到旧的平铺目录。
- 抽出 AI JSON 生成与字段标准化逻辑，避免重复实现解析流程。

### 验收标准

- 后端可正常启动，新增模块能被 Spring 和 MyBatis 扫描。
- 现有 AI 能力不被破坏。
- 新增功能按模块边界落位明确。

## Phase 3 - 角色与背包系统开发

### 目标

- 新增项目级物品库和角色背包系统。
- 接入 AI 物品生成，支持项目物品创建和角色背包定向生成。

### 实施内容

- 新增数据库表 `item` 和 `character_inventory_item`。
- 新增项目物品 CRUD、角色背包 CRUD、AI 物品生成接口。
- 前端在人物管理页提供背包查看、编辑、移除和 AI 生成功能。
- 新增前端类型、API、store 与界面状态管理。

### 验收标准

- 项目内可维护物品定义。
- 角色可维护个人背包。
- AI 可返回结构化物品并进入物品库或背包。

## 实施顺序与依赖

1. 先完成 Phase 1，快速落地可视化入口。
2. 再完成 Phase 2，先整理后端结构，再承接新功能。
3. 最后完成 Phase 3，将角色和背包系统落到新的模块边界内。

## 默认假设

- `cover.png` 首期只用于登录页，不扩散到其他页面。
- 后端保持单体，不拆微服务。
- 物品首期为项目级物品库，不做跨项目共享仓库。
- 背包首期为角色维度，不实现队伍仓库、交易或掉落系统。
- AI 物品生成默认面向中文小说/游戏化叙事设定，输出严格 JSON。
