# 智能体中断恢复与切换协作规则

## 1. 目标

本规则用于解决以下问题：

- 聊天中已经确认的需求和决策只停留在对话里，没有稳定落盘。
- 实现过程中断后，恢复成本高，需要重新翻聊天和代码。
- 切换智能体后，新的执行方无法快速判断当前做到哪里、下一步做什么。

目标是让任意接手的智能体只按固定顺序阅读文档，就能快速恢复上下文并继续执行。

## 2. 固定入口与目录规则

统一使用以下目录结构：

```text
docs/
  agent-context.md
  requirements/
  plans/
  progress/
  archive/
```

规则如下：

- `docs/agent-context.md` 是唯一固定入口文件。
- 中断恢复或切换智能体时，必须先读 `docs/agent-context.md`，不能直接跳进代码。
- `docs/requirements/` 用于按需求拆分需求文档，一份需求一份文档。
- `docs/plans/` 用于按需求拆分实现计划文档，一份需求至少对应一份计划文档。
- `docs/progress/` 用于按需求拆分进度文档，一份需求对应一份持续更新的进度文档。
- `docs/archive/` 用于归档已完成、已废弃或已被替代的文档，可选但推荐使用。

## 3. 文档命名规则

每个需求都必须有稳定的 `Req ID`，并且在需求、计划、进度文档中保持一致。

推荐命名：

- 需求文档：`docs/requirements/REQ-YYYYMMDD-<slug>.md`
- 计划文档：`docs/plans/PLAN-<Req ID>-v1.md`
- 进度文档：`docs/progress/PROGRESS-<Req ID>.md`

约束如下：

- `slug` 使用简短英文或拼音，避免中文文件名频繁变更。
- 如果计划发生重大重写，新增版本号，例如 `v2`，不要直接覆盖旧计划。
- 进度文档原则上持续维护同一个文件，不要每次追加一个新文件。

## 4. 恢复与切换时的固定阅读顺序

中断恢复或切换智能体时，必须严格按下面顺序阅读：

1. `docs/agent-context.md`
2. 对应需求文档
3. 对应最新计划文档
4. 对应进度文档的“当前快照”部分
5. 对应进度文档最近几条关键节点记录
6. 最后再打开代码、SQL、接口定义或测试文件

这样做的原因：

- 先确认当前正在做哪个需求，而不是先猜。
- 先看需求边界，再看实现边界，避免直接根据代码做错误推断。
- 先看当前快照和最近进度，再决定是否需要继续深入代码细节。

## 5. `agent-context.md` 的职责

`docs/agent-context.md` 不是需求详情文档，它只做全局调度和恢复入口。

必须包含以下内容：

- 当前活跃需求列表
- 当前主需求
- 每个需求对应的需求文档、计划文档、进度文档路径
- 每个需求当前状态
- 下一步动作
- 当前阻塞项
- 最近一次更新时间

要求如下：

- 只记录高价值摘要，不复制需求和计划全文。
- 当“当前主需求”变化时，必须立即更新。
- 当“下一步动作”变化时，必须立即更新。
- 当某个需求完成或废弃时，必须更新状态并指向归档位置。

## 6. 需求文档规则

需求文档负责沉淀“聊天里已经确认的内容”，不是实现细节文档。

### 6.1 需求文档必须记录什么

- 需求名称
- `Req ID`
- 当前状态
- 需求来源
- 已确认目标
- 明确范围内事项
- 明确范围外事项
- 已确认约束
- 验收标准
- 待确认问题
- 关键决策记录

### 6.2 什么内容必须写入需求文档

以下内容一旦在聊天中被明确确认，就必须落入需求文档：

- 这个需求到底要做什么
- 这个需求明确不做什么
- 用户已经接受的边界和默认假设
- 用户确认过的接口范围、页面范围、模块范围
- 对旧功能兼容性的要求
- 影响实现方向的重要取舍

### 6.3 需求文档不应该承载什么

- 不要把需求文档写成开发日志。
- 不要把需求文档写成代码实现步骤。
- 不要把需求文档写成只会出现“已完成接口开发”这种模糊结论的状态文档。

### 6.4 需求文档建议结构

```md
# <需求名称>

- Req ID:
- Status:
- Created At:
- Updated At:

## 背景

## 目标

## 范围内

## 范围外

## 约束与默认假设

## 验收标准

## 待确认问题

## 关键决策记录
- [YYYY-MM-DD HH:mm] 决策内容
```

## 7. 计划文档规则

计划文档是需求的具体实现设计，不允许写成空泛任务清单。

### 7.1 计划文档必须回答的问题

- 这个需求具体改哪些模块、文件和数据结构
- 需要新增或修改哪些接口
- 每个接口的字段是什么，哪些必填，哪些选填
- 返回结构是什么
- 前端页面或组件需要怎么接
- 数据库是否需要变更，字段是什么
- 兼容性怎么处理
- 测试怎么验证
- 实施顺序是什么

### 7.2 计划文档严禁出现的写法

下面这些写法不合格：

- “完成技能接口”
- “实现背包功能”
- “对接前端页面”

必须改成更具体的写法，例如：

- `POST /api/projects/{projectId}/skills`，请求体包含 `name`、`type`、`description`、`triggerCondition`、`costDescription`、`cooldownDescription`、`tags`
- 角色技能绑定接口需要 `characterId`、`skillId`、`level`、`source`、`status`、`notes`
- 前端人物页新增“技能标签”和“已掌握技能”两个区域，摘要卡片优先展示结构化技能名称

### 7.3 计划文档必须包含的内容

- 关联 `Req ID`
- 计划版本号
- 实施范围
- 涉及模块与文件
- 数据模型设计
- 接口设计
- 前端交互设计
- 兼容性处理
- 实施步骤
- 每一步的完成标准
- 验证方案
- 风险与回退方案

### 7.4 计划文档建议结构

```md
# <需求名称> 实施计划

- Req ID:
- Plan Version:
- Status:
- Created At:
- Updated At:

## 实施范围

## 涉及模块

## 数据模型

## 接口设计
### 1. <接口名称>
- Method:
- Path:
- Path Params:
- Query Params:
- Request Body:
- Response Body:
- Validation:
- Error Cases:

## 前端交互与状态

## 兼容性要求

## 实施步骤
### Step 1
- 目标:
- 改动:
- 完成标准:

## 验证方案

## 风险与回退
```

## 8. 进度文档规则

进度文档负责记录“计划执行到了哪里”，它是中断恢复和切换智能体时最关键的工作文档。

### 8.1 进度文档必须记录什么

- 当前阶段
- 最近完成的关键节点
- 当前正在处理的内容
- 下一步动作
- 阻塞项
- 已修改文件
- 已完成验证
- 未完成验证
- 风险和遗留事项

### 8.2 什么算关键节点

以下事件发生时，必须写入进度文档：

- 完成一个独立接口或一组强关联接口
- 完成一轮数据库结构调整
- 完成一个前端页面或关键交互区域
- 完成一次重要重构
- 完成一次测试验证
- 发现阻塞问题
- 实施计划被调整
- 准备中断当前工作
- 准备切换到另一个智能体

### 8.3 进度记录必须写到什么粒度

每条关键节点记录至少要写清楚：

- 时间
- 做了什么
- 为什么这么做
- 改了哪些文件
- 是否已验证
- 下一步是什么

不能只写：

- “已完成一部分”
- “处理后端逻辑”
- “更新前端”

### 8.4 进度文档建议结构

```md
# <需求名称> 进度记录

- Req ID:
- Status:
- Created At:
- Updated At:

## 当前快照
- Current Phase:
- Current Task:
- Last Completed:
- Next Action:
- Blockers:
- Latest Verified:
- Latest Unverified:

## 关键节点记录
### [YYYY-MM-DD HH:mm] <节点标题>
- 背景:
- 本次完成:
- 修改文件:
- 验证:
- 风险/遗留:
- 下一步:
```

## 9. 三类文档的更新时机

### 9.1 新需求被确认时

必须执行：

- 新建或更新需求文档
- 在 `agent-context.md` 中登记该需求

### 9.2 进入开发前

必须执行：

- 基于需求文档编写计划文档
- 在 `agent-context.md` 中挂上当前生效的计划文档

### 9.3 开发过程中出现关键节点时

必须执行：

- 更新进度文档
- 如果“下一步动作”或“主需求”变化，同时更新 `agent-context.md`

### 9.4 需求边界变化时

必须执行：

- 先更新需求文档
- 再判断计划文档是否失效
- 如果计划失效，新增计划版本，不直接偷改旧计划的上下文意义
- 在进度文档记录这次变更以及影响

### 9.5 中断前或切换智能体前

必须执行：

- 更新进度文档的“当前快照”
- 明确写出下一步动作
- 明确写出阻塞项和未验证项
- 更新 `agent-context.md`

## 10. 文档之间的职责边界

为了避免三份文档互相污染，职责边界固定如下：

- 需求文档回答“为什么做、做什么、不做什么”
- 计划文档回答“具体怎么做、做到什么粒度、如何验证”
- 进度文档回答“已经做到哪里、下一步做什么、目前卡在哪里”
- `agent-context.md` 回答“当前应该先读哪份文档、哪个需求是主线”

如果一个信息同时出现在三份文档里，通常说明文档边界已经混乱，需要整理。

## 11. 写作质量要求

所有文档必须满足以下要求：

- 用事实描述，不用模糊描述。
- 能写字段就写字段，能写路径就写路径，能写状态就写状态。
- 已完成事项必须尽量附带验证方式。
- 未完成事项必须明确标记为未完成，不允许伪装成已完成。
- 猜测、假设、待确认内容必须单独标记，不能和已确认事实混写。
- 旧决策如果被推翻，必须记录推翻原因，不能静默覆盖。

## 12. 最低执行标准

如果时间紧，至少也必须做到以下最低标准：

1. `agent-context.md` 已更新主需求、下一步动作和文档链接。
2. 需求文档已写清目标、范围、约束和验收标准。
3. 计划文档已写清接口字段、数据结构、实施步骤和验证方法。
4. 进度文档已写清最后完成点、当前状态、下一步和阻塞项。

如果这四点没有满足，就不算具备可靠的中断恢复能力。

## 13. 推荐执行流程

推荐执行顺序如下：

1. 聊天中确认需求后，先落需求文档。
2. 需求边界稳定后，落计划文档。
3. 开始实现后，按关键节点持续更新进度文档。
4. 每次切换主任务、准备中断、准备交接时，更新 `agent-context.md`。
5. 需求完成后，将相关文档状态改为完成，并按需移入 `archive/`。

## 14. 一句话原则

任何一个新接手的智能体，都必须能只通过 `agent-context.md` + 需求文档 + 计划文档 + 进度文档，在不翻完整聊天记录的前提下恢复工作。
